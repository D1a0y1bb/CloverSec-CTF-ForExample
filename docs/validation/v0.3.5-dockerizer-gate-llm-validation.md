# v0.3.5 Dockerizer Gate / LLM Validation

日期：2026-06-14

## 变更

v0.3.5 增加 Dockerizer 改造确认 gate：

- 有源码、Dockerfile、compose 或源码型在线题线索时，不能直接把上游 Dockerfile 当最终交付。
- 未确认 Dockerizer 方案、也没有 `platform_contract_verified=true` 时，批量报告写 `Dockerizer 改造待确认`。
- `failure_cases.jsonl` 记录 `platform_conversion_required`。
- `confirmation_request` 新增 `dockerizer` 动作，用于生成 CONFIG PROPOSAL、风险和确认项。

## LLM 验证

CloudRouter `/v1/models` 可用模型包含：

- `gpt-5.4`
- `gpt-5.4-mini`
- `gpt-5.5`
- `2pt-5.5`

主验收使用 `gpt-5.5`。`gpt-5.4-mini` 只作为字段约束压力测试，不代表真实工程流程推荐模型。

测试场景：

> Web CTF 题目目录包含源码、上游 Dockerfile、docker-compose.yml、端口 8080 和完整 Flag。Dockerizer 方案未确认。

期望输出：

```json
{
  "must_use_dockerizer": true,
  "confirmation_action": "dockerizer",
  "can_archive": false,
  "failure_category": "platform_conversion_required",
  "next_skill": "cloversec-ctf-build-dockerizer",
  "direct_upstream_dockerfile_final": false,
  "xlsx_flag": "flag{demo_full_flag}",
  "hub_auto_submit": false
}
```

结果：

- `gpt-5.5`：通过。
- `gpt-5.4-mini`：严格字段提示下通过；宽松提示下曾把字段写成自然语言，说明 mini 只能作为 schema 检查，不作为主验收。

## 结论

v0.3.5 不再只是记录“题目运行失败”。对容器题和源码型在线题，插件会主动把问题转成 Dockerizer 改造确认流程，并阻断可归档状态。最终是改造、降级验收还是剔除，仍由用户判断。
